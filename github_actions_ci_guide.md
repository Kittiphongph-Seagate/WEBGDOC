# 🚀 คู่มือการทำงานของ GitHub Actions Auto-Tagging CI/CD

ไฟล์นี้เป็นคู่มืออธิบายการทำงานของเวิร์กโฟลว์ (Workflow) ในไฟล์ [auto-tag.yml](file:///C:/Users/761294/Downloads/web-app-fis/WebGDoc/.github/workflows/auto-tag.yml) เพื่ออำนวยความสะดวกในการจัดเก็บประวัติเวอร์ชันของโค้ดแบบอัตโนมัติ

---

## 🎯 จุดประสงค์ของระบบ (Goal)
เพื่อปักป้ายเวอร์ชันย้อนหลัง (Git Release Tag) ของกิ่งที่พัฒนาอยู่โดยอัตโนมัติทุกครั้งที่มีการอัปโหลดโค้ดใหม่ (Push) ขึ้นไปบนระบบ GitHub โดยใช้ฟอร์แมตชื่อป้ายแบบอ่านเข้าใจง่าย:
> **รูปแบบ:** `[ปีเดือนวัน]-[ชื่อ Branch]-[จำนวนครั้งที่รัน]`
> *เช่น* `20260701-gitlab-1` หรือ `20260701-master-12`

---

## 🧩 โครงสร้างการทำงานทีละส่วน (Workflow Breakdown)

โครงสร้างในไฟล์ YAML แบ่งออกเป็น 4 ส่วนหลักดังนี้:

### 1. เงื่อนไขการสั่งให้ทริกเกอร์ทำงาน (Trigger Rules)
```yaml
on:
  push:
    branches:
      - '**'
```
*   **คำอธิบาย:** ระบบจะตรวจจับและสั่งรันงานเมื่อเกิดเหตุการณ์ **Push** ข้อมูลโค้ด
*   **`**`** หมายถึง: ทำงานบน **ทุกกิ่ง (All Branches)** ไม่ว่าจะเป็น `gitlab`, `master`, หรือกิ่งฟีเจอร์อื่นๆ ที่คุณสร้างขึ้นในอนาคต

---

### 2. การกำหนดสิทธิ์ความปลอดภัย (Runner Permissions)
```yaml
permissions:
  contents: write
```
*   **คำอธิบาย:** ตามมาตรฐานความปลอดภัยของ GitHub เวอร์ชันใหม่ เวิร์กโฟลว์จะได้รับสิทธิ์อ่านเท่านั้น (Read-only) เป็นดีฟอลต์
*   **ทำไมต้องเปิด:** เราจำเป็นต้องกำหนดสิทธิ์เขียน **`contents: write`** เพื่อให้บอตของ GitHub (GitHub Actions Token) ได้รับสิทธิ์ในการปักหมุด Tag ใหม่ และ Push ข้อมูล Tag กลับเข้าไปเขียนใน Repository ได้สำเร็จโดยไม่ต้องสร้าง Access Token เอง

---

### 3. โครงสร้างพื้นที่จำลองการประมวลผล (Runner OS)
```yaml
jobs:
  create-tag:
    runs-on: ubuntu-latest
```
*   **คำอธิบาย:** สั่งเปิดใช้เครื่องระบบปฏิบัติการ **Ubuntu Linux** บนระบบคลาวด์ของ GitHub ชั่วคราวเพื่อทำหน้าที่รันคำสั่งเบื้องหลังให้กับเรา (ไม่ต้องสิ้นเปลืองรันบนเครื่องคอมพิวเตอร์ของคุณเอง)

---

### 4. ขั้นตอนการรันคำสั่ง (Execution Steps)

งานย่อยภายในจะรันตามลำดับ 3 ขั้นตอน ดังนี้:

#### **ขั้นตอนที่ A: โคลนโค้ดลงเครื่องจำลอง (Checkout)**
```yaml
- name: Checkout Repository
  uses: actions/checkout@v4
```
*   **คำอธิบาย:** ดาวน์โหลดโค้ดล่าสุดจากกิ่งที่ป้อนเข้ามา นำไปวางในเครื่อง Linux เพื่อเตรียมรันคำสั่ง Git ถัดไป

#### **ขั้นตอนที่ B: ปรับเวลาเป็นเวลาประเทศไทย (Timezone Setup)**
```yaml
- name: Setup Thailand Timezone
  run: sudo timedatectl set-timezone Asia/Bangkok
```
*   **คำอธิบาย:** เครื่องของ GitHub บนระบบคลาวด์จะมีเวลาเริ่มต้นเป็นเวลากลาง (UTC) การสั่งตั้งค่าโซนเวลาเป็น `Asia/Bangkok` จะช่วยให้ตัวแปรเวลา `date` แสดงเวลาประเทศไทยจริงในขณะนั้นได้อย่างถูกต้องตรงกัน

#### **ขั้นตอนที่ C: สร้างและดัน Tag (Generate & Push Tag)**
```yaml
run: |
  # 1. ดึงวันที่ปัจจุบันมาจัดรูปแบบ
  CURRENT_DATE=$(date +'%Y%m%d')
  
  # 2. ปรับแต่งชื่อกิ่ง (ตัดอักษรแปลกปลอมออก เช่น เครื่องหมาย /)
  CLEAN_BRANCH=$(echo "${{ github.ref_name }}" | sed 's/[^a-zA-Z0-9.-]/_/g')
  
  # 3. กำหนดชื่อ Tag รวมสามตัวแปร
  TAG_NAME="${CURRENT_DATE}-${CLEAN_BRANCH}-${{ github.run_number }}"
  echo "กำลังสร้าง Tag: ${TAG_NAME}"
  
  # 4. ระบุตัวตนบอตเพื่อให้ Git อนุญาตให้บันทึก Tag
  git config user.name "github-actions[bot]"
  git config user.email "41898282+github-actions[bot]@users.noreply.github.com"
  
  # 5. สั่งสร้างและ Push
  git tag "$TAG_NAME"
  git push origin "$TAG_NAME"
```

## 🛠️ ความหมายของตัวแปรระบบใน GitHub Actions

| ตัวแปรในคำสั่ง | แหล่งที่มาของข้อมูล | ตัวอย่างค่าที่ได้รับ |
| :--- | :--- | :--- |
| **`$(date +'%Y%m%d')`** | เวลาของระบบปฏิบัติการ Linux | `20260701` |
| **`${{ github.ref_name }}`** | ชื่อกิ่ง (Branch) ที่ทริกเกอร์รัน | `gitlab`, `master` |
| **`${{ github.run_number }}`** | จำนวนครั้งสะสมที่รันบิวต์ (บวกเพิ่มทีละ 1 เสมอ) | `1`, `2`, `14` |
| **`github-actions[bot]`** | บัญชีบอตจำลองระบบของ GitHub | บัญชีผู้เขียนอัตโนมัติ |

## 🔍 วิธีแก้ไขปัญหาเบื้องต้น (Troubleshooting)

หากเกิดเหตุการณ์สร้าง Tag ไม่สำเร็จในอนาคต:
1.  **สิทธิ์ของ Token โดนปิดกั้น:** หากขึ้นข้อความแจ้งเตือน Error `Permission Denied` ใน Log ของ GitHub Actions
    *   *วิธีแก้:* ไปที่หน้าเว็บ GitHub ของคุณ ➡️ **Settings** ➡️ **Actions** ➡️ **General** ➡️ เลื่อนลงไปดูที่หัวข้อ **Workflow permissions** ➡️ เลือกติ๊กที่ **"Read and write permissions"** แล้วกดบันทึก (Save)
2.  **ดูรายงาน Log ย้อนหลัง:**
    *   *วิธีเข้าถึง:* เข้าไปที่แท็บเมนู **Actions** ด้านบนหน้าเว็บ ➡️ คลิกดูประวัติการรันแถวล่าสุด จะแสดงผลการรันเรียงบรรทัดชัดเจน
